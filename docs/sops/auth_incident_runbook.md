## When to Use This SOP

Use this runbook when support receives any of the following symptoms:

- Users are unable to log in and receiving "Invalid credentials" errors
- Users report being locked out of their accounts
- Users are being logged out unexpectedly or their sessions are expiring too quickly
- A user's token is being rejected despite a recent login
- High rate of HTTP 401 or HTTP 403 errors appearing in logs for `/auth/` endpoints
- Alerts for `INTERNAL_AUTH_ERROR` in the structured log stream

## Prerequisites

Before starting this procedure, ensure you have:

- Read access to the application's structured log output (Kibana, CloudWatch, or equivalent)
- Access to the running service environment (Kubernetes namespace or OpenShift project)
- The ability to view or reset environment variables (`JWT_SECRET`, `TOKEN_TTL`, `MAX_LOGIN_ATTEMPTS`)
- Contact details for `@alice` (primary on-call for this service)
- The user's `username` or `user_id` (obtained from the support ticket)

## Step-by-Step Procedure

1. Check the structured logs for the affected username. Filter by `username` and look for `error_code` fields. Note the specific error code ‚Äî each code maps to a distinct procedure below.

2. Identify the error code from the log entry or the HTTP response body returned to the user. The `error_code` field will be one of: `INVALID_CREDENTIALS`, `ACCOUNT_LOCKED`, `TOKEN_EXPIRED`, `TOKEN_INVALID`, or `INTERNAL_AUTH_ERROR`.

3. Follow the Decision Points section below for the identified error code.

4. After applying the fix, ask the affected user to attempt login again and confirm the issue is resolved.

5. If the issue persists after following this runbook, escalate to `@alice` using the Escalation Path below.

6. Document the resolution in the support ticket, including the error code observed, action taken, and outcome.

## Decision Points

### If error code is `INVALID_CREDENTIALS` (HTTP 401)

- **Symptom:** User receives "Invalid username or password" on login.
- **Then:** Verify with the user that they are using the correct username (case-sensitive). Ask them to reset their password via the standard password reset flow. Do not reveal whether the username exists.
- **Else escalate:** If the user is certain of their credentials and the error persists, escalate to `@alice` ‚Äî the user store or credential lookup may be corrupted.

---

### If error code is `ACCOUNT_LOCKED` (HTTP 403)

- **Symptom:** User receives "Account is locked after N failed attempts" on login.
- **Then:** This is triggered automatically after `MAX_LOGIN_ATTEMPTS` consecutive failed logins (default: 5). To unlock, a support engineer with server access must clear the failed-attempt counter for the user by restarting the auth service (this resets the in-memory `_failed_attempts` store). Note the restart window and inform the user when to retry.
- **Production note:** In production this should be a database operation, not a restart. Raise a follow-up ticket if this workaround is being used repeatedly.
- **Else escalate:** If the account shows as locked but the user reports no failed attempts, escalate to `@alice` ‚Äî the counter may have been manipulated or there is a credential-stuffing event in progress.

---

### If error code is `TOKEN_EXPIRED` (HTTP 401)

- **Symptom:** User is logged out unexpectedly or their session ends early.
- **Then:** Check the `TOKEN_TTL` environment variable (default: 3600 seconds = 1 hour) and `REFRESH_TTL` (default: 86400 seconds = 24 hours). If the values have been reduced below expected, restore them to defaults and redeploy. Ask the user to log in again.
- **Else escalate:** If TTL values are correct and tokens are expiring early, escalate to `@alice` ‚Äî there may be a clock skew issue between services or a configuration drift.

---

### If error code is `TOKEN_INVALID` (HTTP 401)

- **Symptom:** A token is being rejected despite the user having logged in recently.
- **Then:** This usually means the `JWT_SECRET` environment variable was rotated, invalidating all existing tokens. All users will need to log in again. Confirm with the platform team (`@genai-autodoc-demo/platform-eng`) whether a secret rotation occurred.
- **Else escalate:** If no secret rotation occurred, escalate to `@alice` ‚Äî a token may have been tampered with or the signing code has a regression.

---

### If error code is `INTERNAL_AUTH_ERROR` (HTTP 500)

- **Symptom:** Users receive a generic server error during login or token operations.
- **Then:** Check the structured logs for the full exception stack trace. Look for connectivity issues to the user store or unexpected `None` values in the token pipeline.
- **Else escalate:** Immediately escalate to `@alice` with the full log trace attached ‚Äî HTTP 500 from auth is a P1 incident.

## Rollback Procedure

If a recent deployment to `auth_service/auth.py` is suspected as the cause:

1. Identify the last known-good commit SHA using `git log --oneline auth_service/auth.py`.
2. Notify `@genai-autodoc-demo/platform-eng` to initiate a rollback deployment to the previous image tag.
3. Verify the rollback by checking `/health` returns `{"status": "ok"}` and attempting a test login.
4. If the rollback resolves the issue, open a post-incident review ticket and assign to `@alice`.
5. If the rollback does not resolve the issue, escalate immediately ‚Äî the problem may be in the environment (secrets, network) rather than the code.
