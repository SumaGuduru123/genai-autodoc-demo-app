## When to Use This SOP

Use this runbook when support receives any of the following symptoms:

- A new user account cannot be created ("email already registered" or "validation error" responses)
- A user account is not found despite the user insisting they are registered
- A user's email or roles cannot be updated
- A user has been deactivated by mistake and needs to be restored
- HTTP 403 errors are returned when calling user management endpoints
- HTTP 422 validation errors are returned for user create or update operations, including errors that list specific disallowed characters in the username
- HTTP 500 `INTERNAL_USER_ERROR` errors appear in the user API logs

## Step-by-Step Procedure

1. Confirm the operation the user or caller was attempting: create, read, update, or delete.

2. Check the HTTP response body for the `error_code` field. It will be one of: `USER_NOT_FOUND`, `DUPLICATE_EMAIL`, `VALIDATION_ERROR`, `PERMISSION_DENIED`, or `INTERNAL_USER_ERROR`.

3. Follow the Decision Points section below for the identified error code.

4. After applying the fix, ask the caller to retry the operation and confirm it succeeds.

5. If the issue persists, escalate to `@alice` using the Escalation Path below.

6. Record the error code, action taken, and outcome in the support ticket.

## Decision Points

### If error code is `USER_NOT_FOUND` (HTTP 404)

- **Symptom:** A `GET /users/{user_id}` or update/delete call returns "User not found".
- **Then:** Verify the `user_id` is correct (it is a UUID v4 string). Confirm the user was not soft-deleted (set to `is_active = False`). Inactive users are excluded from `GET /users` by default ‚Äî try `GET /users?active_only=false` to check.
- **Else escalate:** If the user definitely existed and `active_only=false` returns nothing, escalate to `@alice` ‚Äî the record may have been lost due to a service restart (in-memory store) or a storage issue.

---

### If error code is `DUPLICATE_EMAIL` (HTTP 409)

- **Symptom:** "Email is already registered" error when creating or updating a user.
- **Then:** The email is already associated with another account. Ask the user to provide a different email address, or look up the existing account with that email and determine whether it should be merged or deactivated. Do not create duplicate accounts.
- **Else escalate:** If the email is not visible in any active account, escalate to `@alice` ‚Äî the email index may have a stale entry from a partially completed operation.

---

### If error code is `VALIDATION_ERROR` (HTTP 422)

- **Symptom:** Create or update call fails with a validation error. The error message text indicates which field failed:

  - **Username with disallowed characters:**  
    *"Username 'alice@dev!' contains disallowed character(s): '!', '@'. Only letters (a‚Äìz, A‚ÄìZ), digits (0‚Äì9), underscores (_), and hyphens (-) are permitted."*  
    The error message lists the exact offending characters found in the username. This error is raised by `_validate_username_no_special_chars()` before the length check runs.

  - **Username length error (after character check passes):**  
    *"Username must be 3‚Äì64 characters: letters, digits, _ or -"*  
    The username is either too short (<3 chars) or too long (>64 chars). This error is raised by `_validate_username()` only if the character validation passed.

  - **Email format error:**  
    *"Invalid email address"* ‚Äî email is malformed and does not match RFC-5322 format.

  - **Unknown role(s):**  
    *"Unknown role(s)"* ‚Äî an unrecognised role string was supplied. Allowed values are `viewer`, `developer`, `support`, `ops`, `admin`.

- **Then:** Relay the exact validation message to the caller.

  - For **username character errors**: the error message explicitly lists the disallowed characters (e.g., `'!', '@'`). Ask the caller to remove or replace the listed characters. Only letters (`a-z`, `A-Z`), digits (`0-9`), underscores (`_`), and hyphens (`-`) are permitted. Spaces, `@`, `!`, `#`, `$`, `%`, `.` and all other symbols are rejected.
  
  - For **username length errors**: the username must be 3‚Äì64 characters after special-character validation passes.
  
  - For **email errors**: must be a valid RFC-5322 address.
  
  - For **role errors**: correct the roles list to include only `viewer`, `developer`, `support`, `ops`, `admin`.

- **Else escalate:** If the input looks valid but is still being rejected, escalate to `@alice` ‚Äî the validation regex or logic may have been changed in a recent deployment.

> **Validation flow (current implementation):**  
> Username validation now runs in two steps:  
> 1. Special-character check (`_validate_username_no_special_chars()`) ‚Äî lists offending characters explicitly if any are found.  
> 2. Length check (`_validate_username()` via `_USERNAME_RE`) ‚Äî checks 3‚Äì64 character constraint.  
> If both checks would fail (e.g., `"a@"` has a bad char *and* is too short), the caller sees the **character error first**.

### If error code is `PERMISSION_DENIED` (HTTP 403)

- **Symptom:** A role update or user deletion is rejected with "Only admins may update/delete users".
- **Then:** The caller's JWT does not include the `admin` role. Confirm with the caller whether they should have admin access. If yes, an admin must first update their roles to include `admin` via `PATCH /users/{caller_user_id}`. This action itself requires an admin caller.
- **Else escalate:** If the caller is confirmed as an admin but is still receiving 403, escalate to `@alice` ‚Äî the caller_roles extraction from the JWT may have a regression.

---

### If error code is `INTERNAL_USER_ERROR` (HTTP 500)

- **Symptom:** Any user operation returns a generic server error.
- **Then:** Check the structured logs immediately for the full exception stack trace. Look for unexpected `None` values, UUID generation failures, or storage write failures.
- **Else escalate:** Immediately escalate to `@alice` with the full log trace ‚Äî HTTP 500 from user management is a P1 incident.

## Rollback Procedure

If a recent deployment to `auth_service/user.py` is suspected as the cause:

1. Identify the last known-good commit SHA using `git log --oneline auth_service/user.py`.
2. Notify `@genai-autodoc-demo/platform-eng` to initiate a rollback deployment to the previous image tag.
3. Verify the rollback by calling `GET /users` and confirming it returns a valid response.
4. If the rollback resolves the issue, open a post-incident review ticket assigned to `@alice`.
5. Note that the in-memory user store (`_users`, `_email_index`) is reset on every restart ‚Äî any users created since the last restart will be lost. This is a known limitation of the demo implementation. In production, the store should be backed by a persistent database.
