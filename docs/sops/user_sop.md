## When to Use This SOP

Use this runbook when any of the following conditions are observed in production:

- HTTP `403` responses with error code `PERMISSION_DENIED` ‚Äî a caller is attempting a privileged operation without the `admin` role, or a user account is inactive.
- HTTP `404` responses with error code `USER_NOT_FOUND` ‚Äî a credential check or user lookup fails to locate the account.
- HTTP `409` responses with error code `DUPLICATE_EMAIL` ‚Äî a create or update operation rejects a duplicate email address.
- HTTP `422` responses with error code `VALIDATION_ERROR` ‚Äî malformed username, email, roles input, or a password that contains no special character.
- HTTP `500` responses with error code `INTERNAL_USER_ERROR` ‚Äî unexpected internal error in the user service.

## Step-by-Step Procedure

1. **Identify the error code** from the HTTP response body or structured log entry.
   - Check `error_code` field in the JSON response: `PERMISSION_DENIED`, `USER_NOT_FOUND`, `DUPLICATE_EMAIL`, `VALIDATION_ERROR`, or `INTERNAL_USER_ERROR`.
   - Check `http_status` (403, 404, 409, 422, or 500) to confirm the category.

2. **Locate the relevant log entry** in the application log stream.
   - Filter on `logger = auth_service.user`.
   - Key log messages:
     - `"Local credential check passed"` ‚Üí successful authentication.
     - `"User created"` ‚Üí successful account creation.
     - `"User updated"` ‚Üí successful account update.
     - `"User deactivated"` ‚Üí successful soft-delete.
     - `"Failed to create user"` ‚Üí internal error during `create_user`.
     - `"Failed to update user"` ‚Üí internal error during `update_user`.

3. **For 404 `USER_NOT_FOUND`** ‚Äî confirm the `user_id` or `username` is correct.
   - Ensure `create_user` was called successfully before `verify_user_credentials` or `get_user`.
   - For credential failures: ensure `register_local_credentials` was called with the correct username and user ID before calling `verify_user_credentials`.

4. **For 403 `PERMISSION_DENIED`** ‚Äî determine which operation triggered it.
   - Role update or delete: confirm the caller's session token contains the `admin` role.
   - Account inactive: check `UserRecord.is_active`; re-enable by updating the record if appropriate.

5. **For 409 `DUPLICATE_EMAIL`** ‚Äî have the caller supply a different email address, or identify and deactivate the duplicate account.

6. **For 422 `VALIDATION_ERROR`** ‚Äî return the `message` field to the client. No server-side action is required. Inspect the `detail` field for the specific validation failure:
   - **Username format:** must match `^[a-zA-Z0-9_\-]{3,64}$`.
   - **Email format:** must match RFC-5322 pattern.
   - **Roles:** must be one of `viewer`, `developer`, `support`, `ops`, `admin`.
   - **Password special character:** password supplied to `verify_user_credentials` must contain at least one of `! @ # $ % ^ & * ( ) - _ = + [ ] { } | ; : ' , . < > ? / \` ~ " \`. Advise the client to include a special character and retry.

7. **For 500 `INTERNAL_USER_ERROR`** ‚Äî examine the full exception traceback logged under `exc_info=True` in `create_user` or `update_user`. Identify and fix the root cause before redeploying.

8. **Reproduce and confirm resolution** by attempting the failing operation again after applying the fix.

## Decision Points

### `UserNotFoundError` (HTTP 404 ¬∑ `USER_NOT_FOUND`)
- **Trigger:** `get_user`, `verify_user_credentials`, `delete_user` ‚Äî the `user_id` is not in `_users`, the `username` is not matched, or the password does not match the stored credential.
- **Resolution path:** Confirm the `user_id` or `username` is correct. Ensure `create_user` was called before `verify_user_credentials`. Ensure `register_local_credentials` was called with the correct credentials.

### `PermissionDeniedError` (HTTP 403 ¬∑ `PERMISSION_DENIED`)
- **Trigger 1 (CRUD):** `update_user` ‚Äî caller attempts a role change without `admin` in `caller_roles`. `delete_user` ‚Äî caller is not an admin.
- **Trigger 2 (credential check):** `verify_user_credentials` ‚Äî `UserRecord.is_active` is `False`.
- **Resolution path:** For permission errors on CRUD operations, confirm the API caller's session token contains the correct role claims. For inactive accounts, re-enable the account or advise the user to contact an admin.

### `DuplicateEmailError` (HTTP 409 ¬∑ `DUPLICATE_EMAIL`)
- **Trigger:** `create_user` and `update_user` ‚Äî the supplied email is already present in `_email_index`.
- **Resolution path:** Have the caller supply a different email address, or identify and merge/deactivate the duplicate account.

### `ValidationError` (HTTP 422 ¬∑ `VALIDATION_ERROR`)
- **Trigger:** `_validate_email`, `_validate_username`, `_validate_roles` ‚Äî input does not match the expected format or allowed values. `_validate_password_special_char` ‚Äî the password supplied to `verify_user_credentials` contains no special character (checked via `_SPECIAL_CHAR_RE`).
- **Resolution path:** Return the error message to the client. No server-side action required. Inspect the `detail` field for the specific validation failure. For password failures, advise the client to include at least one special character (e.g. `!`, `@`, `#`, `$`).

### `InternalUserError` (HTTP 500 ¬∑ `INTERNAL_USER_ERROR`)
- **Trigger 1:** `create_user` ‚Äî an unexpected exception is raised after input validation passes.
- **Trigger 2:** `update_user` ‚Äî unexpected exception during record update.
- **Resolution path:** Examine the full exception traceback in the logs. Fix the root cause and redeploy.
