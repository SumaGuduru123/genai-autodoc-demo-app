## When to Use This SOP

Use this SOP when an on-call engineer or support operator receives an alert, customer ticket, or log entry originating from `auth_service/user.py`. Specifically, when any of the following error codes appear in logs or API responses:

- `USER_NOT_FOUND` (HTTP 404)
- `DUPLICATE_EMAIL` (HTTP 409)
- `VALIDATION_ERROR` (HTTP 422)
- `PERMISSION_DENIED` (HTTP 403)
- `INTERNAL_USER_ERROR` (HTTP 500)

This SOP also applies when the user management subsystem produces unexpected `500` errors that do not carry one of the above typed codes.

---

## Prerequisites

Before beginning any investigation:

1. Ensure you have **read access** to application logs (structured JSON logs from `auth_service.user` logger).
2. Confirm you have **read-only database/store access** (or in-memory state dump if running in the current in-memory mode).
3. Have the **affected `user_id`** or **email address** from the ticket or alert.
4. Verify your role allows you to call admin-gated endpoints if remediation is required (`admin` role required for role updates and deletions).
5. Know the environment: **production**, **staging**, or **development** — do not apply destructive remediation steps to production without approval.

## Step-by-Step Procedure

1. **Identify the error code** from the log entry or API response body (`error_code` field).
2. **Look up the affected entity**: search logs for the `user_id` or email address reported in the ticket. Filter by `auth_service.user` logger namespace.
3. **Match the error code to the Decision Points table** below and follow the indicated action.
4. **Attempt remediation** using only the approved API endpoints or admin tooling — never mutate the store directly in production.
5. **Confirm resolution**: re-run the failing operation or ask the customer to retry. Verify the new response is `2xx`.
6. **Log the incident**: record the `user_id`, error code, root cause, action taken, and resolution time in the incident tracker.
7. **Escalate if unresolved** after two remediation attempts — see Escalation Path below.

## Decision Points

### `USER_NOT_FOUND` — HTTP 404 | `UserNotFoundError`
**Trigger:** `get_user()`, `update_user()`, or `delete_user()` was called with a `user_id` that does not exist in `_users`.

| Check | Action |
|---|---|
| Was the `user_id` provided by the client? | Verify the client is sending the correct UUID. Typos and URL-encoding issues are common. |
| Was the user recently deleted (soft-delete)? | Search logs for `"User deactivated"` with the same `user_id`. Soft-deleted users still exist but `is_active=False`. A 404 means the record was never created or was hard-deleted externally. |
| Is this a cross-environment issue? | Confirm the request hit the correct environment. A `user_id` from staging will not exist in production. |
| Still not found after checks? | Escalate — a data-loss event may have occurred. |

---

### `DUPLICATE_EMAIL` — HTTP 409 | `DuplicateEmailError`
**Trigger:** `create_user()` or `update_user()` attempted to use an email address already present in `_email_index`.

| Check | Action |
|---|---|
| Is this a legitimate duplicate registration attempt? | Inform the user the email is already registered and direct them to log in or use password reset. No remediation needed. |
| Is this a data consistency bug (email index out of sync)? | Compare `_email_index` with `_users` records. If `_email_index` contains an email mapped to a non-existent `user_id`, this is a stale index entry — escalate to `@genai-autodoc-demo/security-identity`. |
| Is this during a migration or bulk import? | Pause the import, deduplicate the source data, and retry. |

---

### `VALIDATION_ERROR` — HTTP 422 | `ValidationError`
**Trigger:** `_validate_email()`, `_validate_username()`, or `_validate_roles()` rejected an input value.

| Check | Action |
|---|---|
| Username fails validation | Username must be 3–64 characters, containing only letters (`a-z`, `A-Z`), digits (`0-9`), underscores (`_`), or hyphens (`-`). Ask the user to correct the username. |
| Email fails validation | Email must match standard RFC-5322 format. Ask the user to correct the email address. |
| Roles contain unknown value | Only `viewer`, `developer`, `support`, `ops`, `admin` are allowed. Correct the roles list and retry. |
| Error appears in automated pipeline | Review the pipeline's input sanitisation — it is passing malformed data to the API. |

> **Note:** The previous per-character special-character error listing (`_validate_username_no_special_chars`) has been removed. Username validation errors now produce a single unified message: `"Username must be 3–64 characters: letters, digits, _ or -"`.

---

### `PERMISSION_DENIED` — HTTP 403 | `PermissionDeniedError`
**Trigger:** `update_user()` was called with `roles` parameter but `caller_roles` does not contain `"admin"`, OR `delete_user()` was called without `"admin"` in `caller_roles`.

| Check | Action |
|---|---|
| Is the caller authenticated with the correct role? | Verify the caller's token/session includes the `admin` role. If not, this is expected behaviour — inform the caller they need elevated permissions. |
| Is the role assignment for the caller incorrect? | An admin (`@alice` or `@genai-autodoc-demo/security-identity`) must update the caller's roles via `update_user()` using an admin-scoped session. |
| Is this an automated service account? | Ensure the service account was provisioned with the `admin` role for user-management operations. |

---

### `INTERNAL_USER_ERROR` — HTTP 500 | `InternalUserError`
**Trigger:** An unexpected exception was caught in the `except Exception as exc` handler inside `create_user()` or `update_user()`.

| Check | Action |
|---|---|
| Check logs for `exc_info=True` traceback | The root cause will be logged immediately before the `InternalUserError` is raised. Retrieve the full stack trace from the log entry. |
| Is the in-memory store in a corrupt state? | Inspect `_users` and `_email_index` for consistency. If a partial write occurred (record in `_users` but not `_email_index`), escalate immediately. |
| Is this reproducible? | If reproducible on demand, capture the input payload, attach it to the incident, and escalate to `@genai-autodoc-demo/security-identity`. |
| Is this a one-off transient error? | If not reproducible and logs show no data corruption, monitor for recurrence and document in the incident tracker. |

## Rollback Procedure

> The current implementation uses an **in-memory store**. There is no persistent database to roll back. In a production deployment backed by a database, apply the database-specific rollback steps below.

1. **Identify the last known-good state**: use the `created_at` / `updated_at` timestamps in `UserRecord` to determine when the bad state was introduced.
2. **Soft-undo a bad create**: call `delete_user(user_id, caller_roles=["admin"])` to deactivate the incorrectly created account. This sets `is_active=False` and preserves the audit trail.
3. **Undo a bad role update**: call `update_user(user_id, roles=<previous_roles>, caller_roles=["admin"])` to restore the prior role list.
4. **Undo a bad email update**: call `update_user(user_id, email=<previous_email>, caller_roles=["admin"])` to restore the original email. Verify `_email_index` is consistent afterward.
5. **For persistent DB deployments**: execute the corresponding SQL/transaction rollback script tagged with the deployment identifier and verify record state before re-opening traffic.
6. **Notify affected user**: if PII or account access was impacted, follow the data-breach notification procedure per your organisation's security policy.
